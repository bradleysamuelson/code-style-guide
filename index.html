Coding Style Guide for CSS

Intro and Philosophy

Nobody wants to write ugly code that's difficult to maintain. Those of us who have been building web sites and applications for awhile have all felt the pain of stepping into a project where, for whatever reason, the format and structure was all over the place. I've been a designer and front-end developer for over 15 years, currently a UX Engineer at Oath, and this guide is mostly for my own sanity to try to start new projects off on the right foot. The main focus is to write clean, simple code that's easy for another human to pick up and maintain at any point.

File Structure

If at all possible (and it's always possible), use a CSS preprocessor (preferably SASS). A partial file should be created for each screen/component/whatever makes the most sense. A main file should be used to import the partials, which will give you greater control of the cascade. The main file should contain nothing but imports.

Reset

Browsers are different. Use a reset to normalize them. I recommend Normalize:
https://necolas.github.io/normalize.css/

General Rules (for readability and to avoid common issues)

  • Don't use IDs in selectors (ex: #element)
  • Don't use HTML tags in selectors (ex: div.element)
    • Also avoid adding styles directly to tags (div) unless they're for reset purposes.
  • Avoid nesting for nesting's sake. Only use nesting when increased specificity is needed.
    • Pseudo selectors should be nested. Otherwise, only if it makes sense.
    • If you are intentionally nesting for specificity, don't go deeper than 3 levels (preferably fewer).
  • Don't use !important - there may be rare occasions where it's necessary (overriding bad third-party code comes to mind) but there's almost always a better option.
    • Remember: If everything is !important, nothing will be.
  • Keep things simple.
    • In general, don't go for overly complicated tricks and hacks. The simpler and easier to understand your code is, the easier it will be to maintain.
  • Dont' use margin-top.
    • To avoid undesired outcomes of collapsing margins, instead use padding-top or margin-bottom on the preceding element.
  • Use shorthand properties sparingly and only if you are declaring all values.

Formatting

  • Use 2 spaces for indentation.
  • Always put a space after the colon in declarations.
  • Always put a space before the {.
  • Always put the closing } on a new line with a line break after.
  • When grouping selectors, keep them all on a single line (unless there are too many and it becomes difficult to read).
  • Only write one property per line.
  • Selectors should be lowercase and hyphen-separated.
  • Use the preprocessor's double slash (//) for commenting.
  • Don't include a unit for 0 values.
  • Use a leading 0 for decimal values (ex: 0.25).
  • Hex values should be lowercase.

File and Rule Order

The content of a file should follow the following structure as closely as possible:

  1. Imports
  2. Variables (overrides of global variables or used only in this file)
  3. Mixins (that only apply to this file)
  4. Helper styles
  5. Base styles
  6. Layout styles

Properties should adhere as closely as possible to the following structure:

  1. Rule extensions
  2. Layout or box-model properties
    • Margin, padding, position, display, width, height, etc
  3. Typography
    • Font, line-height, text-decoration, etc
  4. Stylistic properties
    • Color, background, border, animation, etc
  5. Interaction properties
    • Appearance, cursor, pointer-events, etc
  6. Pseudo-selectors
    • :hover, :focus, :active, etc
  7. Pseudo-elements
    • ::before, ::after, ::selection, etc
  8. Modifier classes
  9. Nested elements

Variables

  • Global variables should live in a separate 'variables' file.
  • Local variables should be defined at the top of the file.
  • Always use a variable to define colors.

Nesting

  • In general, try to keep the cascade as flat as possible - avoid nesting more than 3 levels deep.
  • Nesting should be more of a convenience to modify the parent selector rather than targeting for specificity.

Media Queries

  • Media queries should be used within the selector. The idea is to keep relevant styles together rather than create a separate section or file to contain media queries (which often can lead to cascading issues).

.selector {
  float: left;

  @media only screen and (max-width: 767px) {
    float: none;
  }
}

  • Create global variables for frequently used breakpoints.

$screen_sm_max: 767px;

.selector {
  float: left;

  @media only screen and (max-width: $screen_sm_max) {
    float: none;
  }
}

Naming and Separation of Concerns

In code (as in life) I try to stay out of the over-opinionated arguments about which philosophy is better. Blanket statements about the value of completely semantic vs BEM or functional CSS aren't helpful. Instead, my opinion is that the circumstances of the project will dictate the conventions that should be used. My only rule of thumb is to never go too far with functional CSS to the point where you're almost writing inline styles. At some point it starts to feel icky and you'll know it when you see it.
